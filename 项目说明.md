# AblePath

AblePath is a local-first AI desktop assistant for people with limited mobility. The current direction is no longer a Web-only MVP. It is:

- an AI Agent orchestration service,
- an Electron desktop floating assistant for the user,
- a Web dashboard for setup, status, diagnostics, and caregiver configuration,
- a separated caregiver surface for redacted summaries and SOS status.

This README is a handoff document for the next Codex/developer. It should be enough to continue development on another Windows machine without needing the previous conversation.

## Product Direction

AblePath started as a possible rebuild based on `nanoclaw`, but the current decision is to keep AblePath independent:

- Do not couple runtime behavior to `nanoclaw`.
- Do not execute raw model code, pyautogui strings, or arbitrary action text.
- Keep model output behind AblePath's typed `ActionPlan`.
- Treat `UI-TARS-main` and `UI-TARS-desktop-main` as references for GUI agent prompts, action schemas, coordinate handling, and desktop operator design.
- Preserve a safety boundary between planning, preview, confirmation/auto mode, execution, and SOS stop.

The target user experience is a desktop assistant, not a normal web app. The Web UI remains useful for settings, monitoring, diagnostics, and caregiver management.

## Desktop Assistant Behavior

Current desktop shell controls:

- `Listen`: records fixed-duration speech through `/api/listen` and puts recognized text into the command box.
- `Speak`: speaks selected target through `/api/tts`.
- Speak targets: `Auto`, `Command`, `Plan`, `Steps`, `Status`, `Error`.
- `Plan`: calls `/api/agent/command`.
- `Next`: calls `/api/agent/step`.
- `Dry-run`: confirms with `dryRun`.
- `Confirm`: confirms real execution.
- `Stop`: stops the Agent session and cancels pending Auto loop continuation.
- `SOS`: triggers local emergency.
- `screen`: first-plan screenshot context toggle.
- `auto`: automatic confirmation/execution toggle.

Important current semantics:

- `screen` controls whether the first plan includes screen context.
- Auto continuation can force a screenshot even if `screen` was not checked initially.
- Auto loop is bounded to 5 continuation steps per session.
- Stop should remain available during completed-but-continuing Auto loops.

## Current Action Types

The Agent/executor supports these typed actions:

```text
openUrl
openApp
click
doubleClick
type
hotkey
scroll
wait
finished
callUser
```

Windows-specific notes:

- `openUrl` launches GUI processes detached, so Chrome should no longer close after a timeout.
- `openApp` supports installed app launch and special handling for `Recycle Bin`.
- `click` and `doubleClick` use Windows PowerShell/user32 input.
- `type` uses clipboard plus paste for more reliable text entry.
- `hotkey` uses `System.Windows.Forms.SendKeys`.
- `doubleClick` was added because single-clicking desktop icons only selects them.

## Search/Open Website Status

The most recent live issue was:

```text
搜索国科大官网并打开
```

Earlier behavior repeatedly opened new Baidu search tabs instead of clicking the already-visible official result. Phase 62 changed this:

- First `/api/agent/command` may rewrite search-and-open into a Baidu search URL for reliability.
- Follow-up `/api/agent/step` disables that rewrite and preserves screenshot-based click plans.
- Auto continuation forces screenshot capture, so it can click the result even if the first command did not use `screen`.

Expected current behavior:

1. User enters `搜索国科大官网并打开`.
2. First plan opens:

   ```text
   https://www.baidu.com/s?wd=国科大官网
   ```

3. Auto executes it.
4. Auto continuation screenshots the result page.
5. Next plan should click/open the official result, not open another search tab.

Known remaining weakness:

- This is still visually coordinated. A dedicated browser/search-result tool would be more reliable than clicking based on screenshot coordinates.
- A final verification step is still needed before declaring `finished`.

## Known Gaps

High-priority gaps:

- Real browser-page operation is still fragile. The Agent loop exists, but visual result clicking can fail.
- There is no browser-native search-result/open-first-official-result tool yet.
- There is no visible action/result log in the floating shell.
- Stop cancels local pending Auto loop timers, but an in-flight AI planning request cannot be truly aborted server-side yet.
- The loop can still mark success based on low-level input success rather than verified page state.
- True hold-to-talk/release-to-stop recording is not implemented; Listen still uses fixed-duration `/api/listen`.
- Wake-word remains deferred.
- Multi-monitor/high-DPI coordinate normalization needs more real-machine testing.
- `nut-js` remains a future backend option, not the current executor.

Lower-priority gaps:

- Web Control and Screen pages are diagnostics, not the main control entry.
- Caregiver UI is functional but still local/pairing-oriented.
- Remote assist is not implemented.
- Owner auth is local/lightweight, not hardened for network deployment.

## Recommended Next Phase

Recommended Phase 64:

1. Add a browser/search tool instead of relying only on visual clicking.
   - For search-and-open tasks, support an action like `searchWeb` or `openSearchResult`.
   - Use a provider/search endpoint or deterministic search URL + result extraction where possible.
   - Map it back into typed AblePath actions, not raw model code.

2. Add post-action verification.
   - After click/openUrl, screenshot and ask the model whether the goal is complete.
   - Only allow `finished` when the current screenshot proves the requested page/app is open.

3. Add visible desktop execution log.

4. Add server-side cancellation.
